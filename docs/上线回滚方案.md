# 上线回滚方案

使用 Skill：requirements-doc、project-plan

## 1. 回滚触发条件

出现以下任一情况应立即启动回滚：

- 管理端或移动端无法登录。
- 库存、订单、工资、回款金额出现 P1 级错误。
- 数据库迁移执行失败且无法快速修复。
- 前端发布后核心页面白屏或路由不可用。
- Supabase 权限策略导致普通用户越权或无法完成核心作业。

## 2. 前端回滚

1. 保留上一版 `frontend-admin/dist` 和 `frontend-mobile/dist`。
2. 发布失败时，将静态托管目标切回上一版产物。
3. 清理 CDN 或托管平台缓存。
4. 验证 `/login`、`/dashboard`、`/m/login`、`/m/home`。

## 3. 数据库回滚

1. 暂停业务写入。
2. 根据迁移记录确定失败脚本。
3. 优先执行反向 SQL 或恢复上一版函数、策略、字段默认值。
4. 如果数据损坏，按 [备份与恢复说明.md](备份与恢复说明.md) 恢复备份。
5. 回滚后执行库存、订单、工资、财务核验 SQL。

## 4. 环境变量回滚

1. 保留每次发布前的变量快照。
2. 变量变更导致故障时恢复上一版变量。
3. 重新部署前端并清理缓存。
4. 验证 Supabase 连接和文件上传。

## 5. 决策与沟通

| 角色 | 职责 |
| --- | --- |
| 技术负责人 | 判断是否回滚，执行代码和数据库回滚 |
| 运维负责人 | 备份、恢复、环境变量和发布平台操作 |
| 业务负责人 | 确认业务影响和恢复窗口 |
| 项目负责人 | 对外同步进度和最终恢复结论 |

## 6. 回滚后复盘

回滚完成后 24 小时内补充：

- 故障时间线。
- 根因。
- 影响范围。
- 修复方案。
- 防止复发的测试或监控。
