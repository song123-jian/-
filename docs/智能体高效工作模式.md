# 智能体高效工作模式

使用 Skill：requirements-doc、code-generation

更新时间：2026-07-04

## 1. 目标

为 `E:\cx\zs` 内 ERP/MES 程序建立可复用、可审计、可关闭的高效代码实施模式，使需求理解、方案拆分、代码实现、验证和交付文档更新可以并行推进，同时避免重复工作和文件冲突。

本模式适用于管理端 `frontend-admin`、移动端 `frontend-mobile`、数据库脚本 `database`、自动化测试 `tests`、交付文档 `docs` 以及根目录验收标准。

## 2. 基本原则

1. 启动任何方案前，主线程必须完整读取 `C:\Users\宋建\.codex\AGENTS.md`。
2. 只有任务边界清楚、可并行且不会重复扫描同一问题时，才创建智能体。
3. 写入型任务必须明确文件所有权；不同智能体不得同时修改同一文件。
4. 主线程负责最终集成、验证、验收说明和关闭智能体。
5. 获取智能体结果后立即关闭，避免长期占用和状态漂移。
6. 所有文本产物必须 UTF-8 无 BOM，文档抬头优先使用中文。

## 3. 智能体角色

| 角色 | 使用时机 | 输入 | 输出 | 退出条件 |
| --- | --- | --- | --- | --- |
| 交付物审计智能体 | 验收标准、交付包、运维文档需要快速核对时 | `ERP系统交付验收标准.md`、`docs`、`database`、配置文件 | 已满足项、缺口、建议补齐文件 | 输出审计结果后关闭 |
| 测试覆盖审计智能体 | 需要核对核心路径、测试脚本和质量门禁时 | `package.json`、`tests`、前端路由和工具函数 | 覆盖矩阵、缺口、验证建议 | 输出审计结果后关闭 |
| 页面实施智能体 | 单个功能页面需要独立实现时 | 指定页面、相关 API、测试文件 | 限定文件变更和变更报告 | 主线程合并或拒绝结果后关闭 |
| 回归验证智能体 | 主线程实现期间需要并行验证时 | 指定命令、页面路径、截图目录 | 命令结果、截图、失败定位 | 返回验证证据后关闭 |

## 4. 调度生命周期

### 4.1 开始阶段

- 明确任务 ID、目标、输入文件、输出格式、读写范围和禁止操作。
- 判断是否需要拆成小任务；能由主线程一次完成的任务不创建智能体。
- 对只读审计任务使用只读范围；对写入任务指定唯一文件集合。
- 记录启动时间、任务角色和预期退出条件。

### 4.2 过程阶段

- 主线程继续处理不重叠工作，不重复执行智能体正在做的同类审计。
- 智能体只在授权范围内操作；写入型智能体不得修改未分配文件。
- 遇到失败先缩小范围定位原因，再执行最小修复。
- 发生冲突时以主线程最终集成为准，禁止回退用户或其他智能体的无关变更。

### 4.3 结果阶段

- 智能体必须输出变更或审计摘要、证据、风险和建议下一步。
- 主线程审阅结果后执行集成、验证和文档更新。
- 智能体结果一经接收立即关闭。
- 最终交付报告记录关键节点、验证命令、失败项和未完成风险。

## 6. 命令边界

### 6.1 只读审计可用命令

- `rg --files`
- `rg <pattern>`
- `git status --short`
- `git diff -- <path>`
- `node --test --test-only` 仅限已存在且不会写产物的目标测试
- PowerShell 只读文件读取命令

### 6.2 写入和验证命令

- 文件编辑统一由主线程使用 `apply_patch`。
- 目标测试优先运行相关测试文件。
- 全量验证使用 `npm run verify:delivery`。
- 构建命令 `npm run build` 会产生 `dist`，只在需要交付验证时执行。

### 6.3 禁止操作

- 禁止 `git reset --hard`、`git checkout --` 等破坏性回退。
- 禁止删除未明确授权的文件或目录。
- 禁止提交 `.env.local`、密钥、数据库密码、临时日志和个人令牌。
- 禁止在未确认文件所有权时修改同一文件。

## 7. 页面级实施模式

单个功能页面按以下顺序推进：

1. 阅读页面、路由、API、工具函数和已有测试。
2. 梳理本页面业务闭环：输入、校验、提交、跨表影响、输出、异常处理。
3. 修改页面或工具函数，保持现有组件风格。
4. 增加或更新相邻测试。
5. 执行目标测试，必要时执行全量 `npm run verify:delivery`。
6. 页面完成后再进入下一页面，避免跨页面半成品。

## 8. 质量门禁

每个阶段至少通过以下门禁：

| 门禁 | 命令或证据 | 通过标准 |
| --- | --- | --- |
| 文档编码 | `npm run check-encoding` | UTF-8 无 BOM |
| 业务规则 | `npm test` 或目标测试 | 相关用例全部通过 |
| 双端构建 | `npm run build` | 管理端和移动端均构建成功 |
| 交付验证 | `npm run verify:delivery` | 编码、测试、构建一次性通过 |
| 业务证据 | 截图、录屏、命令日志 | 关键路径可追溯 |
| 交付文档 | 项目汇总、验收报告、已知问题 | 与实际结果同步 |

## 9. 失败处理

- 编码失败：定位文件编码或 BOM，修复后重新运行 `npm run check-encoding`。
- 测试失败：先运行单个失败测试，确认业务规则或断言原因，再最小修改。
- 构建失败：优先处理 TypeScript 类型错误和缺失依赖，不做无关重构。
- E2E 或截图失败：记录浏览器、URL、账号、视口、失败步骤和截图路径。
- 生产环境失败：标记为上线阻断项，补充责任人、目标日期和验证人。
