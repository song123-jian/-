# ERP 系统交付验收标准与商业产品对标

版本：V1.0
适用项目：`E:\cx\zs` 注塑厂 ERP/MES 管理系统
验收目标：达到 9.0 分以上企业级交付标准
编制日期：2026-07-04
使用 Skill：requirements-doc、project-plan、api-design、sql-database、frontend-implementation、unit-testing、code-generation

## 1. 文档目标

本文定义本 ERP 系统交付验收的企业级标准、评分模型、业务验收清单、非功能指标、交付物要求、验证证据要求，以及与外部收费 ERP 产品的能力对标要求。

验收结论采用 10 分制：

- 9.0 分及以上：可按企业级生产系统交付，允许存在不影响主流程的低风险问题。
- 8.0-8.9 分：可试运行，不建议正式全量上线，必须形成整改清单。
- 7.0-7.9 分：只可内部演示或有限试点，核心能力存在明显短板。
- 7.0 分以下：不得交付生产使用。

## 3. 企业级验收评分模型

总分 100 分，对应 10 分制。最终得分 = 总分 / 10。验收通过要求总分不低于 90 分，且不得触发硬性否决项。

| 维度 | 权重 | 9.0 分以上标准 |
| --- | ---: | --- |
| 业务闭环完整度 | 24 | 销售、采购入库、生产领料、报工、成品入库、销售出库、回款、工资、费用、对账、库存台账形成闭环 |
| 数据正确性与一致性 | 16 | 库存、订单、发货、回款、工资、财务金额可追溯且可重算，关键表具备约束与防错 |
| 生产制造适配度 | 12 | 覆盖注塑场景的机台、模具、工单、报工、质检、生产领料、成品入库、预警 |
| 仓储与移动作业能力 | 10 | 支持库存查询、批次/库位、台账、盘点、调拨、扫码移动作业 |
| 安全与权限 | 12 | 不泄露密钥，具备认证、角色、关键操作保护、RLS/表级权限策略、日志审计 |
| 性能与稳定性 | 8 | 常用页面可流畅使用，核心查询具备分页和索引，前端构建稳定，关键流程无阻断错误 |
| 易用性与产品完成度 | 6 | 桌面端和移动端布局稳定，空态/错误态/加载态明确，核心操作路径短 |

## 4. 硬性否决项

出现任一情况，即使分数达到 90 分也不得验收通过：

- 数据库初始化脚本无法在 Supabase 干净环境完整执行。
- 管理端或移动端 `npm run build` 失败。
- 登录、库存、销售出库、生产报工、工资结算任一核心链路不可用。
- 前端代码暴露 Supabase service role key、数据库密码或其他生产密钥。
- 普通用户可越权读取或修改不属于其角色的数据。
- 销售出库、生产领料、盘点审批等操作造成库存负数且无业务规则允许。
- 订单金额、回款金额、工资金额、费用金额存在不可解释的差异。
- 无操作日志或关键操作无法追溯到用户、时间、对象。
- 生产环境无备份、无恢复流程、无回滚方案。
- 存在 P0/P1 缺陷未关闭。

## 5. 核心业务验收标准

### 5.1 登录与权限

验收要求：

- 管理端和移动端均可登录、退出、刷新用户信息。
- 未登录访问受保护页面必须跳转到登录页。
- 角色至少区分老板/管理员/操作员等关键身份。
- 密码不得明文存储。
- 登录失败、关键配置变更、业务单据增删改应写入操作日志。

验收证据：

- 登录成功/失败截图。
- 不同角色访问菜单截图。
- 操作日志查询截图。
- 数据库中密码字段非明文证据。

### 5.2 基础资料

验收要求：

- 用户、机台、仓库、库位、模具、产品、客户、供应商可新增、编辑、查询、禁用或删除。
- 产品必须支持原料、半成品、成品分类。
- 机台、模具、产品、仓库编码唯一。
- 产品安全库存、重量、模穴、周期、原料用量等制造参数可维护。
- 客户信用、账期、开票信息可维护。

验收证据：

- 每类基础资料各 3 条新增/编辑/禁用记录。
- 唯一编码重复提交被拦截。
- 关联字段缺失时提示明确。

### 5.3 销售与回款

验收要求：

- 销售订单支持客户、产品、数量、单价、金额、交期、状态。
- 销售订单金额应由明细行自动计算。
- 销售出库后同步更新发货数量和订单状态。
- 回款登记支持按订单登记、校验金额、更新已收金额。
- 应收、已收、未收金额可在订单、回款、对账单中一致体现。

验收证据：

- 创建销售订单并生成明细。
- 部分出库后订单状态变为部分发货。
- 完全出库后订单状态变为已发货。
- 部分回款和完全回款金额一致。
- 对账单金额与订单、回款记录一致。

### 5.4 采购入库与库存

验收要求：

- 采购入库支持产品、仓库、库位、批次、数量、单价、金额。
- 入库后库存数量、库存金额、库存台账同步变化。
- 批次、库位、仓库维度可查询。
- 低库存、临期、过期等风险可形成预警。

验收证据：

- 采购入库前后库存对比截图。
- 库存台账出现对应入库记录。
- 批次维度查询可定位。
- 低库存或临期预警可触发。

### 5.5 生产领料

验收要求：

- 生产订单可领用原材料。
- 领料必须校验库存可用量。
- 领料后库存减少，生产订单领料数量/金额更新。
- 不允许超库存领料，除非配置明确允许。

验收证据：

- 正常领料成功。
- 超库存领料失败并提示原因。
- 领料后库存台账、生产订单字段一致。

### 5.6 生产报工

验收要求：

- 移动端和管理端均可记录报工。
- 报工记录包含工单、人员、机台、模具、工序、班次、良品、不良、开始结束时间。
- 报工后生产订单完成数、良品数、不良数同步更新。
- 计件工资可根据产品/工序/日期匹配计件单价。
- 已结算工资日期不得继续新增或修改影响工资的报工。

验收证据：

- 移动端报工成功截图。
- 管理端报工记录查询截图。
- 生产订单完成数变化截图。
- 工资日汇总金额与报工数量、计件单价一致。

### 5.7 成品入库

验收要求：

- 生产完成后可执行成品入库。
- 入库数量不得超过业务允许的已完成或可入库数量。
- 成品入库后库存增加，生产订单入库数量/金额更新。
- 入库成本或金额可追溯。

验收证据：

- 成品入库单据。
- 库存台账对应记录。
- 生产订单入库数量变化。

### 5.8 销售出库与发货

验收要求：

- 销售出库必须关联销售订单和订单明细。
- 出库产品、客户、订单明细必须匹配。
- 出库数量不得超过订单未发数量。
- 出库后库存扣减、发货单生成、订单明细已发数量更新。

验收证据：

- 正常销售出库成功。
- 超订单数量出库失败。
- 错误产品关联出库失败。
- 发货单、库存台账、销售订单状态一致。

### 5.9 仓库调拨与盘点

验收要求：

- 仓库调拨支持调出仓、调入仓、商品、批次、数量、扫码接收。
- 调拨出库和入库均写入台账。
- 盘点支持创建、开始、录入、提交、审批、驳回。
- 盘点差异应可形成库存调整和差异金额。

验收证据：

- 调拨创建、接收、台账截图。
- 移动端扫码接收截图。
- 盘点审批流截图。
- 账面数、实盘数、差异数计算正确。

### 5.10 质量管理

验收要求：

- 质检记录支持工单、产品、检验类型、结果、不良类型、不良数量、图片、备注。
- 质量看板可展示不良率、不良类型分布、风险提示。
- 不良数量不能超过抽检数量或业务允许范围。

验收证据：

- 合格与不合格质检记录各 1 条。
- 不良分析图表截图。
- 不良数量边界校验截图。

### 5.11 工资管理

验收要求：

- 计件单价支持产品、工序、生效日期、失效日期。
- 同一产品/工序的计价周期不得重叠。
- 日工资根据报工自动汇总。
- 月工资支持汇总、奖惩、结算状态。
- 已结算工资不可被后续报工静默改变。

验收证据：

- 计件价格重叠校验。
- 日工资计算明细。
- 月工资汇总截图。
- 结算后修改拦截截图。

### 5.12 财务与费用

验收要求：

- 费用支出支持类型、金额、日期、收款方、备注。
- 对账单可按客户、月份、订单、回款生成。
- 财务金额与销售订单、回款、费用记录一致。
- 金额保留精度和四舍五入规则明确。

验收证据：

- 费用新增与筛选截图。
- 对账单与订单/回款金额核对表。
- 金额边界校验记录。

### 5.13 报表与预警

验收要求：

- 老板驾驶舱展示销售、生产、库存、质量、财务关键指标。
- 生产看板展示机台状态、工单进度、班次产量、风险。
- 质量看板展示合格率、不良率、不良排行。
- 预警中心覆盖库存、模具、交期、质量等风险。

验收证据：

- 三类看板截图。
- 每个看板数据来源说明。
- 至少 3 类预警触发记录。

### 5.14 系统管理与集成

验收要求：

- 系统配置可查询和更新。
- 操作日志可按用户、模块、动作、时间筛选。
- 集成中心可接收或模拟扫码、电子秤、标签、外部推送等集成数据。
- 云库运维页提供 Supabase 初始化、备份、恢复、环境变量说明。
- 消息中心支持未读数量、已读、批量处理。

验收证据：

- 配置修改记录。
- 日志筛选导出。
- 集成模拟结果。
- 备份恢复演练记录。
- 消息已读状态变化。

### 5.15 注塑专业 1-13 闭环

验收要求：

- 工艺卡支持产品、模具、机台、原料和核心注塑参数维护，启用版本不可被静默覆盖，变更必须保留原因和版本。
- 试模/首件确认必须关联工单、工艺卡、模具、机台和质检结论；首件未允许量产时不得作为开工通过证据。
- 配料/烘料/回料必须校验原料批次、用量、回料比例和烘料参数，并能进入批次追溯链。
- 批次追溯中心可按批次、工单、产品、客户、机台、模具正查和反查销售、生产、领料、报工、质检、入库、出库证据。
- 开工准备/齐套检查必须覆盖原料、模具、机台、工艺卡、首件、人员/班次；不通过项必须形成待办或安灯异常。
- 设备维修必须支持故障上报、分派、维修中、完成、关闭，并为机台状态联动预留数据。
- 模具寿命/保养看板必须根据模次、保养周期、寿命和异常次数识别正常、待保养、逾期风险。
- 安灯异常中心必须覆盖缺料、首件驳回、设备故障、质量异常、齐套不通过，支持分派、处理、关闭。
- 包装/标签/条码中心必须支持产品码、批次码、箱标、客户标签模板和打印任务数据。
- 客诉/8D 必须关联客户、产品、批次和 D1-D8 改善证据，关闭前需保留纠正预防措施。
- OEE 必须按机台、班次、日期计算稼动率、性能效率、良品率、OEE、停机和换模损失。
- 工艺变更必须覆盖工艺卡、BOM、原料替代、模具变更版本，记录审批状态、生效时间和变更原因。
- 请购单必须承接 BOM/MRP 缺料结果，记录缺料量、请购量、建议供应商、期望到货和采购状态。

验收证据：

- 管理端 13 个入口按业务归属分布到生产管理、质量管理、仓库管理、报表中心；真实路径仍保持 `/injection/*`，并均能打开、查询、新增、编辑、删除和执行可用状态动作。
- 移动端 13 个入口均能打开并提交现场表单，首页/待办中心能显示注塑待办。
- `database/init.sql` 与 `database/supabase-cloud.sql` 包含注塑专业表组和索引。
- `npm test` 包含并通过 4 个注塑专项测试文件。
- 浏览器抽检无空白页、无控制台阻断错误，表单必填与边界校验可见。

| Microsoft Dynamics 365 Supply Chain/Business Central | 覆盖供应链计划、采购、车间、订单、仓库、资产管理；仓库与制造、质量、采购、调拨、销售等流程集成 | 仓储能力要与销售、生产、质量联动，不应是孤立库存表 | 仓库调拨、盘点、销售出库、生产领料、成品入库都必须联动台账 |
| Odoo Manufacturing/Inventory/Quality | 支持制造订单、计划、加工、车间平板、维护、质量反馈、条码和 IoT/MES 场景 | 移动端和车间作业体验是制造 ERP 产品化关键 | 移动端报工、质检、库存、盘点、调拨必须可独立完成核心作业 |
| 金蝶云·星空 | 面向中型企业，覆盖财务、供应链、制造、PLM、全渠道，支持多组织、多语言、多时区 | 国内企业级 ERP 强调业财协同、多组织、制造深度 | 单工厂 9.0 目标需完成业财协同；多组织、PLM、全渠道可作为后续增强 |
| 用友 U9 Cloud | 面向中大型制造业，强调财务业务一体化、多组织集团化、SRM/WMS/CRM 等集成 | 企业级交付需具备集成扩展和集团化演进空间 | 当前系统集成中心、Supabase 架构和数据模型需保留扩展接口与审计能力 |

### 6.1 对标结论

本系统不是通用大型集团 ERP，也不应以 SAP、NetSuite、Dynamics 365 的完整平台能力作为一次性交付边界。9.0 分企业级验收应定位为：

- 在注塑厂单组织或小型多车间场景内，达到收费 ERP 的核心业务闭环体验。
- 在生产、库存、质量、工资、移动报工等细分场景上，达到比通用 ERP 更贴近现场的可用性。
- 在高级计划、BOM/MRP、成本会计、集团合并、多语言多账套、AI 预测等方面，作为 9.5 分以上产品化增强项，不作为本次 9.0 分硬性范围。

## 8. 非功能验收标准

### 8.1 性能

| 指标 | 验收标准 |
| --- | --- |
| 管理端首屏 | 生产环境常规网络下 LCP 不超过 2.5 秒 |
| 交互响应 | 关键按钮、表单、筛选 INP 不超过 200ms |
| 布局稳定 | CLS 不超过 0.1 |
| 列表查询 | 1 万条以内业务数据分页查询 P95 不超过 1.5 秒 |
| 看板加载 | 老板驾驶舱、生产看板、质量看板 P95 不超过 3 秒 |
| 移动端作业 | 报工、质检、盘点、调拨提交 P95 不超过 2 秒 |

### 8.2 安全

验收标准参考 OWASP ASVS 和 OWASP Top 10：

- 前端不得包含 service role key、数据库密码、管理后台私钥。
- Supabase 表必须按生产角色配置 RLS 或等价权限控制。
- 用户只能访问其角色授权的数据和功能。
- 文件上传必须限制 bucket、路径、文件类型、大小和公开访问范围。
- 关键接口必须校验输入，禁止 SQL 注入、越权、任意文件上传。
- 认证 token 必须安全存储和过期处理。
- 所有 P0/P1 安全问题必须关闭。

### 8.3 可靠性

| 指标 | 验收标准 |
| --- | --- |
| 可用性 | 生产环境月可用性目标不低于 99.5% |
| 备份 | 每日自动备份，关键上线前手动备份 |
| RPO | 常规生产目标不超过 24 小时 |
| RTO | 常规生产目标不超过 4 小时 |
| 回滚 | 前端版本、数据库脚本、环境变量均有回滚说明 |
| 数据一致性 | 库存、订单、回款、工资、费用关键数据可重算 |

### 8.4 兼容性

- 桌面端：Chrome、Edge 最新两个稳定版本。
- 移动端：主流 Android Chrome、iOS Safari。
- 分辨率：管理端覆盖 1366x768、1440x900、1920x1080；移动端覆盖 375x667、390x844、414x896。
- 所有按钮文字、表格列、弹窗表单在目标分辨率下不得重叠或溢出。

### 8.5 可维护性

- 业务工具函数有单元测试覆盖。
- API 适配层保持统一返回结构。
- 数据字段命名前后端转换规则明确。
- 页面、工具函数、测试文件命名和模块边界一致。
- 新增业务规则必须同步更新测试和验收说明。

## 9. 测试与验证要求

### 9.1 必跑命令

```bash
npm run verify:delivery
npm run check-encoding
npm test
npm run build
```

通过标准：

- `verify:delivery` 可一次性完成编码、自动化测试和双端构建验证。
- `check-encoding` 无 UTF-8/BOM 异常。
- `npm test` 通过率 100%，不得存在失败用例。
- `npm run build` 管理端和移动端均成功。

### 9.2 核心回归路径

至少完成以下 20 条核心路径，并保留截图或录屏：

1. 管理端登录、刷新、退出。
2. 移动端登录、首页进入、退出。
3. 新增产品、客户、供应商、仓库、机台、模具。
4. 创建销售订单并维护明细。
5. 采购入库后库存增加、台账生成。
6. 创建生产订单。
7. 生产领料后库存减少、工单领料数量更新。
8. 移动端报工后工单完成数和工资日汇总更新。
9. 移动端质检录入后质量看板更新。
10. 成品入库后库存增加、工单入库数量更新。
11. 销售出库后发货单生成、订单状态更新。
12. 回款登记后应收/已收/未收金额更新。
13. 仓库调拨创建、移动端接收、库存变化。
14. 盘点创建、录入、提交、审批、差异处理。
15. 计件单价生效、过期、重叠校验。
16. 月工资汇总、奖惩、结算。
17. 费用支出新增、筛选、统计。
18. 对账单按客户/月生成并核对金额。
19. 老板驾驶舱、生产看板、质量看板加载。
20. 系统日志、配置、消息、云库运维页面可用。

### 9.3 缺陷分级

| 等级 | 定义 | 验收要求 |
| --- | --- | --- |
| P0 | 系统不可用、数据丢失、严重安全泄露 | 必须 0 个 |
| P1 | 核心流程阻断、金额/库存错误、权限绕过 | 必须 0 个 |
| P2 | 重要功能异常但有替代路径 | 不超过 3 个，需有明确修复计划 |
| P3 | 文案、样式、低频边界问题 | 不超过 10 个，纳入迭代 |

缺陷闭环必须记录编号、等级、状态、责任人、目标关闭日期、验证人、验证证据和是否阻断上线。涉及 RLS 权限、备份恢复、库存、金额、工资结算的缺陷，在正式生产上线前必须关闭或由业务负责人书面确认降级。

| 角色 | 必验场景 | 通过标准 |
| --- | --- | --- |
| 老板/经营负责人 | 老板驾驶舱、生产看板、质量看板、经营预警 | 指标可读、来源明确、刷新无阻断 |

| 财务/工资 | 回款、费用、对账、计件工资、月工资 | 金额可核对，结算后不可被静默篡改 |

### 10.4 数据迁移与初始数据验收

上线前必须核对以下初始数据：

- 用户、角色、组织、仓库、库位、机台、模具、产品、客户、供应商。
- 产品分类、单位、重量、模穴、生产周期、安全库存、原料用量。
- 期初库存数量、批次、库位、库存金额。
- 计件价格、有效期、工资结算起始月份。
- 未完成销售订单、生产订单、回款、费用和对账期初。

数据迁移通过标准：导入总量、抽样明细、金额合计、库存合计均可对账；关键主数据编码唯一；失败数据必须有重导记录或业务确认。

### 10.5 运维交接与支持 SLA

正式交付必须明确：

- 运维负责人、业务负责人、技术负责人和紧急联系人。
- 日常巡检频率：工作日检查登录、核心页面、数据库容量、备份状态、错误日志。
- 故障响应：P0 30 分钟内响应，P1 2 小时内响应，P2 1 个工作日内响应，P3 纳入迭代。
- 监控告警：数据库连接异常、备份失败、前端构建版本异常、接口持续失败、库存/金额异常波动。
- 变更流程：生产变更必须有版本号、变更说明、验证记录、回滚方案和发布负责人。
