# 注塑厂管理系统重构落地技术文档

## 1. 文档目的

本文档用于指导当前注塑厂管理系统做一次全量重构，目标不是“修修补补”，而是把系统升级成一套可持续演进、可追溯、可对账、可移动作业的工厂管理平台。

当前项目已经具备 Spring Boot 后端、Vue 3 管理端、移动端、数据库脚本和备份能力，适合在现有代码基础上做结构重整，而不是推倒重来。

## 2. 重构目标

1. 统一业务主链路：销售、生产、质检、库存、财务、工资、报表一套口径。
2. 统一数据底座：全部切换到 PostgreSQL / Supabase，移除 MariaDB 依赖。
3. 统一单据模型：主表、明细表、流水表、日志表四层结构。
4. 统一前端组织：管理端按业务域分组，移动端按现场动作聚焦。
5. 统一追溯能力：任意一笔入库、出库、报工、质检、发货都能反查源单。
6. 统一运维方式：备份、恢复、日志、通知、权限、任务调度标准化。

## 3. 参考原则

这里参考的是成熟 ERP / MES 的共性设计，不是照搬某个产品界面。

1. 业务模块要分域，不要按页面堆菜单。
2. 单据流要明确状态机，不允许随意跳状态。
3. 库存必须以流水为准，不靠人工改数量。
4. 移动端优先做高频动作，低频配置留在 PC 端。
5. 现场网络不稳定时，系统要能离线缓存并补传。
6. 所有统计都要从明细和流水汇总，不要手填报表。

## 5. 目标架构

### 5.1 技术栈

- 后端：Spring Boot 3.x + MyBatis-Plus + Spring Security + JWT
- 数据库：Supabase PostgreSQL
- 缓存：无，统一采用 Spring Boot + PostgreSQL
- 前端管理端：Vue 3 + Vite + Element Plus
- 前端移动端：Vue 3 + Vant
- 报表：ECharts
- 导出：EasyExcel
- 二维码：ZXing

### 5.2 组织方式

系统保持单体部署，但在代码层面按业务域拆分：

1. `base` 基础资料域
2. `prod` 生产执行域
3. `stock` 库存物流域
4. `qc` 质量追溯域
5. `sale` 销售交付域
6. `salary` 工资计件域
7. `finance` 财务结算域
8. `system` 平台运维域

## 6. 业务域设计

### 6.1 基础资料域

目标是把所有单据依赖的主数据收口。

核心对象：

- 用户
- 角色
- 客户
- 供应商
- 产品
- 模具
- 机台
- 仓库
- 库位
- 班次

关键要求：

- 编码统一
- 状态统一
- 软删除统一
- 审计字段统一

### 6.2 生产执行域

生产域是系统的主干。

核心对象：

- 生产订单
- 报工记录
- 停机记录
- 上下模记录
- 设备点检记录
- 模具保养记录
- 预警记录

状态建议：

- 草稿
- 已下达
- 生产中
- 待质检
- 已完工
- 已关闭

### 6.3 库存物流域

库存不再只维护一个“当前数量”，而是由流水计算。

核心对象：

- 材料批次
- 库存流水
- 入库单
- 出库单
- 调拨单
- 盘点单
- 库存台账

### 6.4 质量追溯域

核心对象：

- 质检记录
- 缺陷分析
- 抽检记录
- 不良原因
- 留样记录

目标：

- 产品能追到订单
- 订单能追到报工
- 报工能追到机台、模具、班次
- 库存能追到来源和去向

### 6.5 销售交付域

核心对象：

- 销售订单
- 发货单
- 回款记录
- 交付状态

### 6.6 工资计件域

核心对象：

- 计件单价
- 日工资
- 月工资汇总
- 奖惩调整

### 6.7 财务结算域

核心对象：

- 费用支出
- 应收应付
- 对账单
- 财务汇总

### 6.8 平台运维域

核心对象：

- 系统配置
- 操作日志
- 消息通知
- 数据备份
- 集成配置
- 定时任务

## 7. 数据模型重构

### 7.1 基础规范

每张业务表统一保留以下字段：

- `id`
- `code`
- `status`
- `created_at`
- `updated_at`
- `created_by`
- `updated_by`
- `deleted`
- `remark`

### 7.2 三层数据结构

1. 主表：业务头信息
2. 明细表：业务条目
3. 流水表：所有会改变库存、工资、财务状态的动作

### 7.3 推荐的关键表组

- `supplier`
- `product`
- `machine`
- `mold`
- `warehouse`
- `warehouse_location`
- `prod_order`
- `prod_report`
- `downtime_record`
- `mold_mount_record`
- `mold_maintenance_record`
- `machine_inspection_record`
- `qc_record`
- `material_batch`
- `stock`
- `stock_move`
- `stock_transfer`
- `stock_transfer_item`
- `stock_inventory`
- `stock_inventory_item`
- `sale_order`
- `sale_order_item`
- `delivery_order`
- `delivery_order_item`
- `payment_record`
- `expense_record`
- `salary_daily`
- `salary_adjust`
- `notification`
- `sys_operation_log`

### 7.4 PostgreSQL 落库要求

- 日期和时间统一使用 `timestamp`
- 金额统一使用 `numeric`
- 枚举优先使用字典表或字符串码值
- 批量写入要考虑事务和幂等
- 索引重点放在：
  - 单号
  - 状态
  - 业务日期
  - 机台
  - 模具
  - 产品
  - 仓库

## 8. 接口设计规范

### 8.1 统一风格

每个业务域至少保持以下接口：

- `list` 分页查询
- `get` 单据详情
- `create` 新增
- `update` 修改
- `delete` 删除
- `submit` 提交
- `approve` 审核
- `revoke` 反审核
- `void` 作废

### 8.2 返回结构

统一使用现有 `R` 包装对象，补齐以下信息：

- `code`
- `message`
- `data`
- `traceId`

### 8.3 业务接口原则

- Controller 只接参数和返回值
- Service 负责编排业务
- Mapper 只负责数据访问
- 复杂查询要拆 DTO / VO
- 所有单据必须保留操作日志

## 10. 重构实施路线

### 阶段 0：底座校正

目标：

- 彻底移除 MariaDB 依赖
- PostgreSQL / Supabase 可用
- 备份恢复可用

交付物：

- 数据源配置统一
- `pg_dump` / `psql` 备份恢复脚本
- 初始化脚本调整为 PostgreSQL

验收：

- 后端能启动
- 数据库能连通
- 备份恢复能跑通

### 阶段 1：基础资料收口

目标：

- 所有业务主数据统一入口
- 菜单和页面结构稳定

交付物：

- 用户、客户、供应商、产品、机台、模具、仓库、库位的统一 CRUD
- 前端菜单按域整理

验收：

- 所有下拉选项都来自主数据
- 无重复维护逻辑

### 阶段 2：生产主链路重构

目标：

- 订单、报工、停机、上下模、点检、保养形成闭环

交付物：

- 生产订单状态机
- 报工单据
- 停机记录
- 模具保养
- 设备点检
- 预警中心

验收：

- 一张生产单能串起机台、模具、班次、产量、不良

### 阶段 3：库存和质量闭环

目标：

- 库存流向清楚
- 质量追溯完整

交付物：

- 库存流水
- 入库 / 出库 / 调拨 / 盘点
- 质检记录
- 不良分析

验收：

- 任意库存变动可追溯源单
- 任意成品可反查生产和质检记录

### 阶段 4：销售、工资、财务打通

目标：

- 数据口径统一
- 老板驾驶舱可直接出数

交付物：

- 销售订单、发货、回款
- 计件工资、日工资、月汇总
- 费用、对账、财务汇总

验收：

- 应收、应付、工资、利润能从流水汇总得到

### 阶段 5：移动端和现场补强

目标：

- 现场使用更轻
- 断网可用

交付物：

- 扫码报工
- 移动质检
- 离线队列
- 任务提醒

验收：

- 网络恢复后可自动补传

## 11. 迁移方案

### 11.1 数据迁移

1. 导出旧库数据。
2. 清洗字段类型和编码。
3. 先迁主数据，再迁单据，再迁流水。
4. 抽样对账。
5. 上线前冻结老库写入。

### 11.2 灰度切换

推荐顺序：

1. 先切基础资料
2. 再切生产报工
3. 再切库存
4. 再切质检和财务

### 11.3 回滚策略

- 每个阶段保留数据库备份
- 每个模块保留旧接口适配窗口
- 上线前做一次全量回放测试

## 12. 风险与控制

### 12.1 主要风险

- 数据类型迁移不完整
- 状态机不统一
- 库存流水缺索引导致慢查询
- 离线补传冲突
- 权限范围过大

### 12.2 控制措施

- 数据表逐张核对
- 每个单据都加状态流转测试
- 核心表先建索引再上线
- 离线队列必须有幂等键
- 权限按岗位分层

## 13. 验收标准

1. 登录、主数据、生产、库存、质检、财务、报表都可正常使用。
2. PostgreSQL 云库与本地环境行为一致。
3. 任意单据都能查看来源、去向和操作日志。
4. 管理端和移动端都能独立完成各自核心任务。
5. 备份、恢复、导出、通知、日志可用。
6. MariaDB 相关配置和脚本完全退出主流程。

## 14. 建议排期

1. 第 1 周：底座、数据库、备份恢复。
2. 第 2 周：基础资料、权限、菜单重组。
3. 第 3 周：生产主链路。
4. 第 4 周：库存、质检、追溯。
5. 第 5 周：销售、工资、财务。
6. 第 6 周：移动端、离线、看板、联调。

## 15. 结论

这次重构的核心不是“换个数据库”或者“重写几个页面”，而是把系统从“能用”升级成“能长期用”。

最优落地路径是：

1. 单体架构不拆。
2. PostgreSQL / Supabase 先落地。
3. 业务按域重组。
4. 单据、流水、追溯、权限统一。
5. 先闭环核心流程，再补细节体验。

这样做，最稳，也最适合当前这套注塑厂管理系统。
