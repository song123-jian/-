# 注塑厂管理系统技术文档

## 当前架构

本项目已调整为前端直连 Supabase 云数据库的架构，仓库内不再维护 Java/Spring Boot 后端工程。

- 管理端：`frontend-admin`
- 移动端：`frontend-mobile`
- 数据库：Supabase PostgreSQL
- 文件存储：Supabase Storage
- 登录能力：优先使用 Supabase Auth；也支持数据库 RPC `erp_login`
- 本地运行：仅启动前端开发服务器，不再启动任何本地后端进程

## 运行配置

两个前端都需要在各自目录下创建本地环境文件：

```bash
cd frontend-admin
cp .env.example .env.local

cd ../frontend-mobile
cp .env.example .env.local
```

环境变量：

```bash
VITE_SUPABASE_URL=https://your-project-ref.supabase.co
VITE_SUPABASE_ANON_KEY=your-supabase-anon-key
VITE_SUPABASE_AUTH_EMAIL_DOMAIN=inject-erp.example.com
VITE_SUPABASE_STORAGE_BUCKET=erp-files
```

## 数据库初始化

1. 在 Supabase 项目 SQL Editor 中执行 `database/init.sql`，创建 ERP/MES 业务表。
2. 执行 `database/supabase-cloud.sql`，创建登录 RPC、Storage bucket 和浏览器直连所需权限。
3. 推荐在 Supabase Auth 中创建用户。若用户名不是邮箱，前端会按以下规则转换：

```text
用户名 -> 用户名@VITE_SUPABASE_AUTH_EMAIL_DOMAIN
```

例如 `songjian` 会转换为 `songjian@inject-erp.example.com`。

## 前端数据访问

`frontend-admin/src/api/index.ts` 与 `frontend-mobile/src/api/index.ts` 已替换为 Supabase 请求适配层。现有页面仍然调用原来的 `api/*.ts` 函数，但底层不再走 `/api` HTTP 代理，而是使用 `@supabase/supabase-js` 直接访问表、RPC 和 Storage。

字段转换在请求层统一处理：

- 写入：camelCase 自动转换为 snake_case
- 读取：snake_case 自动转换为 camelCase
- 分页：返回 `data.records`、`data.list` 和 `data.total`

## 本地启动

根目录快捷命令：

```bash
npm install
npm run dev:admin
npm run dev:mobile
```

也可以分别进入子项目启动：

```bash
cd frontend-admin
npm install
npm run dev

cd frontend-mobile
npm install
npm run dev
```

默认访问地址：

```text
管理端: http://127.0.0.1:3000
移动端: http://127.0.0.1:3001
```

## 迁移收口说明

- 仓库中的 `backend` Java 后端源码已移除，前端不再依赖 `/api` HTTP 代理。
- 系统管理中的“数据备份”页面已改为 Supabase 云库运维说明页，不再伪造本地备份/恢复动作。
- 开发环境下若页面提示未配置 Supabase，请优先检查：

```bash
frontend-admin/.env.local
frontend-mobile/.env.local
database/init.sql
database/supabase-cloud.sql
```

## 重要约束

- 仓库不再包含后端服务；不要恢复 `backend` 目录或 `/api` 代理。
- 生产环境必须在 Supabase 中配置合适的 RLS 策略。当前脚本提供的是迁移可用的基础策略，后续应按角色细化表级权限。
- `.env.local` 只保存在本地，不提交仓库。
